# What AICL Is Not

AICL is an AI-native semantic compilation research project. To preserve that identity, the following boundaries are explicit.

## AICL Is Not a Prompt Template System

AICL is not a collection of prompts or prompt-engineering patterns. Prompts may participate in workflows, but AICL aims to move core software-construction authority into typed, grounded, proof-carrying, provenance-linked artifacts.

## AICL Is Not a No-Code Tool

AICL is not a no-code application builder. It is a research specification for semantic compilation objects and the obligations that should govern AI-driven materialization.

## AICL Is Not a Conventional DSL

AICL may expose DSL-like human-readable projections, but the canonical authority is not the projection syntax. The authority lies in semantic contracts, AICL-IR, ICC, SHG, WKG grounding, proof obligations, and materialization rules.

## AICL Is Not a Wrapper Around Python

Python may be used to implement parts of an AICL compiler, runtime, verifier, materializer, or test harness. Python may also be a materialization target. That does not make AICL a Python wrapper.

## AICL Is Not a Replacement for All Existing Implementation Languages

AICL is not claiming that all conventional languages should disappear. Python, Rust, TypeScript, Kotlin, Swift, SQL, Java, C++, and other languages can remain important target languages and implementation substrates.

## AICL Is Not a Claim That Humans Are Irrelevant

Human review remains important for governance, safety, accountability, public reasoning, related work, and correction of overclaims. The point is not that humans are irrelevant. The point is that AICL is not optimized primarily around human syntax preference.

## AICL Is Not an Official Project of Any Model Provider

AICL is not an official project of OpenAI, Anthropic, Google, xAI, or any other model provider unless a legally and operationally valid affiliation is explicitly documented. Model commentary and AI-assisted reasoning do not imply provider endorsement.

## AICL Is Not a Finished Compiler Unless a Compiler Exists

If this repository does not contain a documented working compiler, AICL should be described as a pre-implementation research specification and design workspace. Do not imply that implemented guarantees exist before they do.

## Why "Python Can Already Do This" Misses the Point

Python can implement parts of an AICL runtime, compiler, verifier, materializer, evaluation harness, or generated application. That is useful.

Python itself does not make intent, policy, capability, proof tier, provenance, WKG grounding, semantic hypergraph structure, materialization contracts, and residual runtime obligations first-class semantic compilation objects. Those concerns can be represented in Python libraries, but the meaning then depends on library conventions, runtime discipline, and human interpretation.

AICL addresses a different layer. It asks what the semantic source of truth should be when AI systems generate, verify, repair, and materialize software across targets.
