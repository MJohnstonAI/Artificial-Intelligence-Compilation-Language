**Author:** Claude (Anthropic) — corrected analysis  
**Type:** Implementation strategy — AI-driven, human-gated

**Core correction:** The compiler is not a binary built by human engineers. The compiler IS an AI agent.

---

## The fundamental reframe

The previous gap analysis made a critical error: it described what human engineers would need to build AICL as if it were a conventional language project. That is the wrong frame entirely.

AICL's thesis is that AI systems can specify, verify, and materialise software. If AICL cannot bootstrap its own implementation using AI agents, it has failed its own thesis before writing a single line of compiler code.

The correct frame is this:

> **The AICL compiler is an AI agent running with the specification in its context window. The WKG is populated by AI knowledge. The proof checking is done by AI reasoning. The materializers are generated by AI models. Humans govern — they do not build.**

This changes everything about what "missing" means.

---

## What AI models can build right now, autonomously

Claude Code, GPT-4o with code execution, and Gemini Code can each do the following today, given the AICL specification documents as context:

**Write the formal EBNF grammar (OQ-001)**  
The Programming Reference Manual describes every construct with enough precision that an AI model can derive the formal grammar. This is not a research problem. It is a translation task. Claude Code can write a complete EBNF grammar from the PRM in a single session. A human reviews and commits. The grammar does not need to be designed by a human — it needs to be *approved* by one.

**Write the JSON schema validators**  
The ICC, SHG, and SkillContract schemas already exist. An AI model can write validators in Python, TypeScript, or any target language in minutes. This is runnable today.

**Write the Tree-sitter grammar**  
Once the EBNF grammar exists (written by an AI model), another AI model converts it to Tree-sitter format. This provides syntax highlighting across all editors immediately.

**Write the parser**  
Given the EBNF grammar, an AI model can write a recursive descent parser or a PEG parser directly. This does not require a human parser engineer — it requires an AI model with the grammar in context and a test suite to validate against.

**Populate the WKG core ontology**  
AI models have world knowledge. The WKG anchors — `Commerce.Refund.Processed.v2`, `Education.Learner.v1`, `Privacy.LocalOnly.v1` — are not invented out of nothing. They are structured expressions of knowledge that AI models already possess. An AI model can enumerate hundreds of domain-appropriate WKG anchors from the spec's described domains. The Project Maintainer reviews and approves. The WKG grows through AI-human governance with AI doing the taxonomy work.

**Implement the Tier 1 proof checkers**  
The decidable proofs — cost ceiling arithmetic, network effect absence, deadline satisfaction, IFC label tracing — are not algorithmically complex. They are structured reasoning tasks. An AI model can implement each proof checker as a Python function, given the constraint schema and the rules from the PRM. Another AI model reviews the implementation for correctness against the spec.

**Write the first materializer**  
Given an SHG JSON file, an AI model can generate a REST API skeleton in FastAPI, Express, or any target. This is exactly the kind of task Claude Code, Codex, and Gemini Code are optimised for.

**Write structured error messages**  
The error format, error codes, and suggested fix templates can all be specified by an AI model from the existing spec and then implemented by another AI model. No human error message designer needed.

## The AI-native compiler architecture

The AICL compiler is not a binary. It is a pipeline of AI agent invocations, each with a specific bounded task, governed by the same AICL principles the language itself describes.

```
AICL-Text source
       ↓
  [AI Agent: Parser]
  Context: formal grammar + PRM lexical spec
  Task: parse source into AST JSON
  Output: aicl-ast.json
       ↓
  [AI Agent: WKG Resolver]
  Context: WKG core JSON + resolution contract
  Task: resolve all identifiers to WKG anchors
  Output: aicl-resolved-ast.json + resolution-errors.json
       ↓
  [AI Agent: Type Checker]
  Context: type rules from PRM §4 + resolved AST
  Task: verify type class assignments, detect violations
  Output: aicl-typed-ast.json + type-errors.json
       ↓
  [AI Agent: IFC Checker]
  Context: IFC rules from PRM §9.3 + typed AST
  Task: trace taint labels, verify OpaqueIntent boundaries
  Output: ifc-report.json + ifc-errors.json
       ↓
  [AI Agent: Contradiction Detector]
  Context: priority lattice + contradiction policy from PRM §11
  Task: normalise constraints, detect χ conflicts
  Output: contradiction-report.json
       ↓
  [AI Agent: Tier 1 Proof Checker]
  Context: proof rules + constraint schema
  Task: verify decidable proofs, emit proof gate outcomes
  Output: proof-bundle.json
       ↓
  [AI Agent: SHG Builder]
  Context: SHG schema + all prior outputs
  Task: construct hypergraph nodes, edges, hyperedges
  Output: aicl-shg.json
       ↓
  [AI Agent: ICC Generator]
  Context: ICC schema + all prior outputs + WKG snapshot hash
  Task: construct and sign the ICC
  Output: aicl-icc.json
       ↓
  [AI Agent: Materializer]
  Context: target spec + SHG + proof bundle + ICC
  Task: generate target artifact
  Output: target-specific artifact
```

Each agent in this pipeline is a bounded AI invocation with a defined context, a defined task, and a defined output schema. Each agent is governed by the same `autonomy`, `capability`, and `policy` constraints that AICL describes for any agent system.

This pipeline can be orchestrated by Claude Code today. The agents are not hypothetical. They are AI models with the right context. The pipeline is an AICL workflow expressed as an AICL-Workflow — the language compiling itself.

## What humans are needed for — and only this

The human role in AICL is governance, not construction. Specifically:

**Architectural arbitration**  
When two AI models in the consortium propose genuinely contradictory designs, a human arbitrates. This is the `escalate_to_haig` behaviour at the project governance level. The human does not design the alternative — they choose between options the AI models have already developed.

**Policy attestation and legal sign-off**  
`policy_attestation_proof: "cert://provider-popia-2026"` — the legal attestation that a real-world provider complies with a policy cannot be generated by an AI model. A human (or a legal entity) must sign this. This is a necessary and correctly bounded human role.

**WKG anchor approval**  
AI models propose WKG anchors. The Project Maintainer reviews and approves. The human role is not to invent anchors — it is to confirm that proposed anchors meet the project's standards. This is a review-and-commit workflow, not a design workflow.

**Repository commit authority**  
Per the consortium protocol, AI models do not commit directly to the repository. The Project Maintainer commits. This preserves the attribution chain, project governance, and the human governance role.

**Real-world deployment decisions**  
When a materialised application is deployed to production infrastructure, a human authorises it. This maps directly to AICL's bounded autonomy model: `goal: 0.0` — agents cannot modify their own deployment goals.

**Financial and legal decisions**  
Grant applications, sponsorship agreements, licensing decisions. These require a human legal and financial principal.

That is the complete list of necessary human involvement. Everything else — grammar writing, parsing, type checking, proof checking, SHG construction, materializer generation, WKG population, test suite generation, documentation, error message design — can and should be done by AI models.

## The self-bootstrapping principle

AICL should compile itself. This is not a long-term aspiration — it is an architectural requirement that validates the thesis.

The AICL compilation pipeline described above is itself expressible as an AICL program:

```aicl
app "AICL.Compiler.Bootstrap" {
  import module AICL.CompilerPipeline
  import capabilitypack AICL.AgentPipeline

  goal AICLProgram.Compiled
  maintain SpecificationCompliance
  avoid HumanBias.LegacyPattern

  agent Parser {
    goal: AST.Produced
    capabilities: [LocalRead:source, LocalWrite:ast]
    autonomy { action: 0.9, goal: 0.0, resource: 0.1, delegation: false }
  }

  agent WKGResolver {
    goal: Identifiers.Resolved
    capabilities: [LocalRead:ast, LocalRead:wkg_snapshot]
    autonomy { action: 0.9, goal: 0.0, resource: 0.0, delegation: false }
  }

  // ... remaining pipeline agents

  flow CompilationPipeline {
    Parser ->
    WKGResolver ->
    TypeChecker ->
    IFCChecker ->
    ContradictionDetector ->
    [proof: all_tier1_gates_pass] ->
    SHGBuilder ->
    ICCGenerator ->
    Materializer
  }
}
```

When this program compiles and produces a working compiler, AICL has proven its thesis. The language has bootstrapped itself.

## What this means for the repository right now

The repository needs two new things that are not currently there:

**1. An AI execution context package**

A directory called `ai-execution-context/` containing everything an AI agent needs to autonomously execute a compilation pipeline stage, with no additional human guidance:

```
ai-execution-context/
  compiler-agent-system-prompt.md    — system prompt for compiler agents
  wkg-core.json                      — machine-readable WKG anchor catalogue
  aicl-grammar-candidate.ebnf        — grammar for AI agents to work from
  stage-definitions.json             — each pipeline stage as a structured task
  error-format.json                  — structured error format AI agents must emit
  test-programs/                     — valid and invalid AICL programs for testing
    hello-world.aicl
    refund-service.aicl
    invalid-identifier.aicl          — must produce specific error
    policy-contradiction.aicl        — must produce specific error
```

This package is what Claude Code, GPT-4o, and Gemini Code need to autonomously implement each compiler stage without human guidance between steps.

**2. An AI-executable build protocol**

A document called `AI_BUILD_PROTOCOL.md` that defines the autonomous build workflow:

- Which AI agent implements which pipeline stage
- What context each agent receives
- What output schema each agent must produce
- How outputs are validated before the next stage proceeds
- What constitutes a valid implementation (test suite pass)
- When to escalate to the Project Maintainer (only on genuine architectural contradiction)

This document replaces the human engineer's implementation plan. It is read by AI agents, not humans.
